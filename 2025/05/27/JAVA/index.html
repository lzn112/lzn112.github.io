JVM & JRE & JDK

  • JDK（Java Development Kit）是 Java 开发工具包，JRE + javac + javadoc（文档生成器）+jdb（调试器）+jconsole（监控工具）+javap（反编译工具)…

  • JRE 是运行已编译 Java 程序所需的环境
    - JVM : 也就是我们上面提到的 Java 虚拟机
    - Java 基础类库（Class Library）：一组标准的类库，

  • JIT（Just in Time Compilation）

当 JIT 编译器完成第一次编译后，其会将字节码对应的机器码保存下来，下次可以直接使用。而我们知道，机器码的运行效率肯定是高于 Java 解释器的。这也解释了我们为什么经常会说 Java 是编译与解释共存的语言 。

成员变量/局部变量

变量

成员变量 vs 局部变量

  • 成员变量可以被 public,private,static 等修饰符所修饰，局部变量不行

  • 俩者都可以用final 修饰

  • static 修饰的变量 属于类 ， 放在方法区或元空间，没有static 属于实例对象的，放在堆里

  • 局部变量不逃逸出方法区放在栈里，

  • 成员变量不赋值的话，会自动赋值（JVM在分配对象内存时，会自动零值初始化）

  • 类型 默认值
    int 0
    long 0L
    float 0.0f
    double 0.0d
    char \u0000
    boolean false
    引用类型 null

  • 局部变量必须赋初值

static

  • 静态变量属于类，一般通过类访问，经常搭配final
  • 静态方法可以通过类访问，因此不应该能调用非静态成员

方法

  • 发生在同一个类中（或者父类和子类之间），方法名必须相同，参数类型不同、个数不同、顺序不同，方法返回值和访问修饰符可以不同。

  • 重写

    • 方法名、参数列表必须相同，子类方法返回值类型应比父类方法返回值类型更小或相等，抛出的异常范围小于等于父类，访问修饰符范围大于等于父类。
    • 如果父类方法访问修饰符为 private/final/static 则子类就不能重写该方法，但是被 static 修饰的方法能够被再次声明。
    • 构造方法无法被重写。

  • 可变长参数（方法重载中，优先匹配固定参数的，变长参数编译后会变成数组）

对象

对象实例 vs 对象引用

  • 对象的相等一般比较的是内存中存放的内容是否相等。
  • 引用相等一般比较的是他们指向的内存地址是否相等。
  • 对比项 构造方法 静态工厂方法
    是否关键字支持 Java 语言特性，必须用 new 调用 普通静态方法，调用时无 new
    方法名是否固定 必须与类名相同 任意命名，如 create, of, from
    是否可以返回子类对象 ❌ 固定返回当前类实例 ✅ 可以返回任意类型（灵活）
    是否支持缓存/单例 ❌ 不方便实现 ✅ 可封装缓存、池、单例等
    是否能隐藏实现细节 ❌ 直接暴露构造过程 ✅ 可内部隐藏构造逻辑
    设计模式关联 构造器初始化 常用于工厂模式（Factory Method）
    是否能重命名 ❌ 必须类名一致 ✅ 方法名灵活，自定义语义
  • 对比项 接口 (interface) 抽象类 (abstract class)
    关键字 interface abstract class
    是否可被实例化 ❌ 不可 ❌ 不可
    方法类型 默认都是 public abstract（可有 default static Java 9+ private） 可以是 public、protected、private 等任意
    是否可有变量 只能有 public static final 常量 可以有成员变量
    是否有构造方法 ❌ 无构造方法 ✅ 可以定义构造方法（子类中super会调用）
    支持多继承吗 ✅ 支持实现多个接口 ❌ 不支持继承多个抽象类（Java 单继承）
    是否用于规范 ✅ 纯规范化接口 ✅ 也可用于抽象规范 + 部分实现
    是否可有默认实现 ✅（Java 8+）可有 default 方法 ✅ 支持具体方法实现
    多态支持 强（接口是典型的多态入口） 强（也可以通过抽象父类引用调用子类）
    目的 行为规范（所作行为） 代码复用（隶属关系）
    成员变量 public static final 任意
  • 比较项 浅拷贝 深拷贝
    基本对象复制 ✅ 会复制 ✅ 会复制
    引用对象复制 ❌ 只复制引用 ✅ 复制引用所指的实际对象（递归）
    是否完全独立 ❌ 改变引用对象会互相影响 ✅ 改变不会影响原始对象
    实现复杂度 简单，直接 super.clone() 复杂，需要递归处理所有引用字段
    性能开销 低 高（尤其嵌套层级多时）
    应用场景 原型设计模式、不重要的字段引用 数据隔离、深层对象状态防篡改

String

String vs StringBuffer vs StringBuilder

  • StringBuilder 与 StringBuffer 都继承自 AbstractStringBuilder 类
  • String 是不可变常量，线程安全
  • StringBuffer 加了同步锁，所以也线程安全 StringBuilder 没加 线程不安全
  • 每次对 String 类型进行改变的时候，都会生成一个新的 String 对象，然后将指针指向新的 String 对象。StringBuffer 每次都会对 StringBuffer 对象本身进行操作，而不是生成新的对象并改变对象引用。相同情况下使用 StringBuilder 相比使用 StringBuffer 仅能获得 10%~15% 左右的性能提升，但却要冒多线程不安全的风险

String不可变(https://www.zhihu.com/question/20618891/answer/114125846)

  • public final class :这个final 表示不能被子类化

  • private final char ：🔒 表示 value 是一个引用类型常量：这个引用只能指向一个特定的 char[] 数组，一旦指向，就不能更改指向。

  • 不会暴露 value[] 的引用

  • 字符串对象通过“+”的字符串拼接方式，实际上是通过 StringBuilder 调用 append() 方法实现的，拼接完成之后调用 toString() 得到一个 String 对象
